# Phishing Analysis Practical Guide: Step-by-Step Security Analyst Manual

This guide serves as a practical, technical manual for security analysts and IT professionals to systematically investigate suspicious emails, extract indicators of compromise (IoCs), and mitigate organizational threats safely.

---

## Phase 1: Secure Email Collection & Environment Isolation

Analyzing a phishing email from a standard, active mail client carries inherent risks, such as accidental link activation or payload execution. To prevent incident escalation:

1.  **Do Not Open Links or Attachments**: Never click any hyperlinks, images, or files in the active mail preview.
2.  **Extract the Raw Source Code**:
    *   **Microsoft Outlook**: Double-click the email -> File -> Properties -> View "Internet Headers". Or, File -> Save As -> Select `Text Only` or `.eml`.
    *   **Gmail**: Open the email -> Click the three dots (More) -> Select "Show original" -> Click "Download Original" (saves as `.eml`).
    *   **Microsoft Office 365 (Web)**: Open the email -> Click the three dots -> View -> View message details.
3.  **Transfer to an Isolated Sandbox**: Place the raw source text into a dedicated analysis environment (e.g., an offline virtual machine, or a secure, non-rendered text editor like VS Code or Notepad++). Rename the file extension to `.txt` to prevent OS-level double-click triggers.

## Phase 2: Deciphering SMTP Email Headers

Email headers tell the true story of a message's routing and authenticity. They are read from **bottom to top** (oldest routing hop to newest).

```
[Target Mail Server]  <--- (Receives and appends final "Received" header)
        ^
        |
[Intermediary Gateways]
        ^
        |
[Originating Server]  <--- (First "Received" header - records threat actor IP!)
```

### 1. The Core Fields
*   **From**: The display name and email shown to the user (highly vulnerable to spoofing).
*   **Return-Path** (or `Envelope-From`): The actual address where bounce-back/undelivered notifications are sent. A mismatch between `From` and `Return-Path` is a primary indicator of phishing.
*   **Message-ID**: A unique identifier generated by the originating mail server. Check if the domain in `<...@[domain]>` matches the sending domain. Mismatches suggest automated scripting or spoofing.

### 2. Investigating the "Received" Hops
Each server that handles the email appends a `Received` line at the top. The **lowest** `Received` header in the source represents the originating server.
*   **Format**: `Received: from [Sender hostname/IP] by [Receiver hostname] with [Protocol] id [ID] for <[Target]>; [Timestamp]`
*   **Analysis Action**: Extract the originating IP address from the lowest hop and conduct a reverse-DNS lookup or threat intelligence query (e.g., using VirusTotal or AbuseIPDB).

### 3. Email Authentication Standards
Modern mail servers inspect cryptographic and structural keys to verify authenticity. Analyze these indicators in the `Authentication-Results` or `Received-SPF` headers:

| Standard | Full Name | How it Works | Security Indicator |
|:---|:---|:---|:---|
| **SPF** | Sender Policy Framework | A DNS TXT record listing all IP addresses authorized to send emails on behalf of a specific domain. | Check `Received-SPF: pass` or `fail`. If `fail`, the sending server IP is not authorized by the claimed domain. |
| **DKIM** | DomainKeys Identified Mail | An cryptographic signature (private key on sender server, public key in DNS) embedded in email headers to verify that the message body was not altered in transit. | Check `dkim=pass` in the headers. A fail indicates tampering or signature forgery. |
| **DMARC** | Domain-based Message Authentication, Reporting, and Conformance | A policy layer instructing receiving mail servers on how to handle emails that fail SPF or DKIM checks (options: `none`, `quarantine`, `reject`). | Look for `dmarc=pass` or `fail`. A domain with a `p=reject` policy will block unauthorized attempts, while `p=none` allows them through for observation. |

## Phase 3: Open Source Intelligence (OSINT) on Domains

Phishers register domains that mimic legitimate entities (known as **Typosquatting** or **homoglyph attacks**).

### 1. Common Domain Typosquatting Tactics
*   **Homoglyphs/Character Substitution**: Replacing letters with visually similar numbers or characters (e.g., `paypa1-support.com` using `1` instead of `l`, or `micros0ft.com` using `0` instead of `o`).
*   **Subdomain Manipulation**: Using a legitimate brand name as a subdomain on an unrelated domain (e.g., `paypal.com.secure-update-alert.net` - the real domain is `secure-update-alert.net`).
*   **Alternative TLDs**: Registering a familiar brand name under a cheap, high-risk Top-Level Domain (e.g., `targetcorp-security.xyz` instead of `.com`).

### 2. Technical Investigation Checklist
When evaluating a domain (e.g., `paypa1-support.com`), utilize public OSINT web tools:

1.  **WHOIS Lookup**:
    *   **Action**: Query WHOIS data (using tools like `whois.domaintools.com` or `mxtoolbox.com`).
    *   **Indicators**: Look for **Creation Date** (recently registered domains < 30 days old are high risk), **Registrar** (cheap registrars like Namecheap or registrar privacy proxies are common for phishing), and **Registrant Country** (mismatches with the physical brand location).
2.  **DNS Records**:
    *   **Action**: Perform an `A` (IPv4) or `MX` (Mail Exchanger) record query.
    *   **Indicators**: Lack of an MX record suggests the domain is only used to host a landing page, not standard communications.
3.  **Threat Intel Repositories**:
    *   **Action**: Submit the domain to **VirusTotal**, **URLVoid**, or **Cisco Talos Intelligence**.
    *   **Indicators**: Checking community comments and current blacklist flags to see if the infrastructure has been reported in active campaigns.

## Phase 4: Safe Hyperlink and URL Examination

Malicious links are often obfuscated to bypass email filters and trick human eyes.

### 1. Decoding URL Obfuscation Techniques
Before analyzing a link, it must be normalized:
*   **URL Encoding**: Characters replaced with `%` followed by their hex value (e.g., `%41` represents `A`). Decodes can reveal hidden target destinations.
*   **Base64/Binary Strings**: Look for lengthy, random-looking character strings in query parameters (e.g., `?id=ZW1wbG95ZWUAdGFyZ2V0LWNvcnAuY29t`). These often encode target usernames to pre-fill phishing forms.
*   **Link Shorteners**: Use of public services (e.g., `bit.ly`, `tinyurl.com`) to hide the ultimate landing page destination.

### 2. Safe Link Analysis Workflow
> [!WARNING]
> NEVER visit a suspicious link on your host machine or standard corporate browser. This can trigger active exploits, log active telemetry (alerting the attacker that your address is "live"), or drop malware payloads.

1.  **Defang the URL**: When documenting links, replace protocol tags to prevent accidental clicks:
    *   Original: `https://paypa1-support.com/secure-login`
    *   Defanged: `hxxps[://]paypa1-support[.]com/secure-login`
2.  **Simulated URL Expansion**: Use services like `wheregoes.com` or `expandurl.net` to map the redirect hops of a shortened link without loading active scripts.
3.  **Security Sandboxing Tools**:
    *   **VirusTotal URL Scan**: Submit the link to verify community and vendor ratings.
    *   **urlscan.io**: A powerful, free tool that visits the website using a sandboxed browser, captures a physical screenshot of the landing page, and details all DOM trees, scripts loaded, and connections established.

## Phase 5: Handling Malicious Attachments

Phishing emails containing attachments (like `Lottery_Winning_Certificate_2026.zip` in Example 02) aim to deliver malicious payloads directly.

### 1. High-Risk File Extensions
*   **Archived Formats**: `.zip`, `.rar`, `.7z`, `.iso` (used to bypass scanning filters).
*   **Executables/Scripts**: `.exe`, `.scr`, `.bat`, `.ps1`, `.lnk`, `.vbs` (runs malicious code directly).
*   **Office Macros**: `.docm`, `.xlsm` (executes Visual Basic macros upon opening, downloading secondary stages).
*   **PDFs/HTML files**: `.pdf` (may contain embedded redirect links), `.html` (often holds encoded local credential harvesting pages, i.e., "HTML smuggling").

### 2. Analyst Triage Steps
1.  **Extract File Hash**: Obtain the cryptographic hash (SHA-256 or MD5) of the attachment.
    *   *Windows PowerShell Command*: `Get-FileHash .\path_to_attachment -Algorithm SHA256`
2.  **Database Lookup**: Search the SHA-256 hash on VirusTotal. If the hash is already known, it will return full analysis results immediately without uploading the file.
3.  **Execute in Sandbox**: If the file is unknown, run it in an isolated malware analysis sandbox (e.g., **Hybrid Analysis**, **Any.Run**, or **Cuckoo Sandbox**) to analyze its behavioral heuristics (processes spawned, network connections, file edits).

## Phase 6: Operational Threat Reporting & Incident Response

Once an email is verified as malicious, the analyst must compile an actionable incident report for the Security Operations Center (SOC).

### Actionable Indicators of Compromise (IoCs) Template
A professional reporting email or ticket should isolate indicators to facilitate rapid firewall/endpoint blocklists:

```text
[INCIDENT REPORT] Phishing Campaign Triage: [Campaign Name / Theme]
===================================================================
Triage Classification: HIGH / CRITICAL / MEDIUM
Attack Type: Credential Harvesting / BEC / Malware Downloader

[KEY THREAT INDICATORS]
Sender Address: security-update-alert@paypa1-support.com
Sender IP: 198.51.100.42 (Originating Host: mail.paypa1-support.com)
Sender Country: [Country Name]
Phishing URL: hxxps[://]paypa1-support[.]com/secure-login/login.php
Malicious Domain: paypa1-support[.]com
Attachment Name: Lottery_Winning_Certificate_2026.zip
Attachment Hash (SHA-256): e3b0c44298fc1c149afbf4c8996fb92427ae41e4649b934ca495991b7852b855

[IMMEDIATE MITIGATION ACTIONS RECOMMENDED]
1. Domain Blocklist: Inject "paypa1-support.com" into the Secure Email Gateway (SEG) and DNS Firewall.
2. IP Blocklist: Null-route traffic from 198.51.100.42 at the perimeter firewall.
3. Account Audit: Scan Exchange/Office 365 logs for any other recipients of messages matching Message-ID headers or sender domains.
4. Host Mitigation: Purge the email from all active user mailboxes matching the campaign template.
```
